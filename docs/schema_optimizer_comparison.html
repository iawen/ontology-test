🎯 一、背景与目标

已有能力：extract_ontology.py 通过数据库表/字段结构 + LLM 提取，已构建出基础 Ontology（Class、Metrics、Relationships、Concepts）。

新增需求：利用业务方提供的文档（Word/Excel/PDF），进一步优化已有的 Ontology 资产——重点是：

  • Class：优化 name_cn / description / 业务边界（排除 fields 部分，fields 由数据库结构确定）
  • Metrics：补充 description / category / calculation / formula / dimensions / required_dimensions / filters_hint / chart_type
  • Relationships：补充/修正 source / target / type / join_key
  • Concepts：补充树状层级（subject_domain / dimension_group / fact_group）

两个版本（v1、v2）采用了不同的策略来实现这一目标，下面逐一分析。

📦 二、v1 架构分析（单次全量模式）

核心策略

v1 采用"一次性全量送入"策略：把所有未审核的 Class、Metrics、Relationships、Concepts 和所有文档内容一次性塞进一个巨大的 LLM prompt，让 LLM 在一次调用中完成所有优化。

关键代码路径

# v1 核心流程
docs_context = _build_document_context(docs)  # 拼接所有文档，上限 45000 字符
all_assets = _load_all_assets(scenario_id)    # 加载所有未审核资产
prompt = OPTIMIZATION_PROMPT.format(
    docs=docs_context,
    assets=json.dumps(all_assets, ...),       # 全量资产
    ...
)
result = await client.chat.completions.create(prompt)  # 单次调用
_apply_optimization(scenario_id, result)     # 直接写回 DB

优势

  • 全局视角 LLM 能看到所有资产和所有文档，做跨表关系推导时上下文完整
  • 实现简单 单次 LLM 调用，代码逻辑清晰，约 587 行
  • 一致性高 所有资产在同一上下文中优化，命名风格/分类标准一致
  • 速度快 只需 1 次 LLM 调用（如果不超过上下文窗口的话）

劣势

  • 上下文爆炸 当 Class > 15 或文档 > 3 个时，prompt 轻松超过 32K token，LLM 输出被截断
  • JSON 截断风险 单次输出大量 JSON，尾部资产（通常是 Concepts）极易被丢弃
  • 无重试机制 JSON 解析失败直接抛异常，整个优化任务失败
  • 无增量能力 每次都是全量重跑，无法只优化新增资产
  • 成本高 每次调用都送入全量上下文，token 消耗大

🚀 三、v2 架构分析（分批关联模式）

核心策略

v2 采用"按 Class 分批关联"策略：以 Class 为锚点，把每个 Class 及其关联的 Metrics、Relationships、Concepts 打包成批次（每批 ≤6 Class / ≤15 Metrics），分多次 LLM 调用完成优化。

关键代码路径

# v2 核心流程
batches = _build_batches(classes, metrics, relationships, concepts,
                         max_classes=6, max_metrics=15)
for batch in batches:
    prompt = OPTIMIZATION_PROMPT.format(
        docs=docs_context,
        assets=json.dumps(batch, ...),    # 分批资产
        ...
    )
    result = await client.chat.completions.create(prompt)
    all_results.extend(result["assets"])  # 累积结果

_apply_optimization(scenario_id, {"assets": all_results})

分批算法亮点

  • 以 Class 为锚点：每个批次围绕 1-6 个 Class，把关联的 Metrics/Relationships/Concepts 一起打包
  • 孤立资产兜底：没有关联到任何 Class 的 Metrics/Relationships/Concepts 单独打包处理
  • 动态批次大小：BATCH_MAX_CLASSES=6, BATCH_MAX_METRICS=15，可配置

优势

  • 上下文安全 每批 prompt 大小可控，不会超过上下文窗口
  • JSON 不易截断 每批输出量小，JSON 完整性有保障
  • 可扩展 支持大量 Class/文档场景，资产越多优势越明显
  • 关联性保持 以 Class 为锚点打包，保持了资产间的语义关联
  • 容错性好 单批失败不影响其他批次（如果实现了 per-batch try-catch）

劣势

  • 文件不完整 v2 文件在 _extract_json 函数处被截断，缺少 LLM 调用、结果合并、DB 写入等核心逻辑
  • 全局视角缺失 分批后 LLM 看不到全部资产，跨表关系推导能力下降
  • 一致性风险 不同批次可能产生命名风格/分类标准不一致
  • 成本可能更高 多次调用 × 每次都带文档上下文 = 总 token 可能超过 v1
  • 文档重复传输 每批都带完整文档上下文，文档内容被重复传输 N 次

🏆 五、综合评判

结论：v2 架构方向正确，但需补全 + 改进

v2 的分批策略在架构思路上明显优于 v1——它解决了 v1 最致命的"上下文爆炸"和"JSON 截断"问题，是面向生产环境的正确方向。

但 v2 存在两个关键问题：

  1. 文件不完整：在 _extract_json 处截断，缺少 LLM 调用、结果合并、DB 写入等核心逻辑，无法直接运行
  2. 全局视角缺失：分批后 LLM 看不到全部资产，跨表关系推导和命名一致性无法保证

建议：以 v2 的分批策略为基础，融合 v1 的全局视角优势，并引入下文提出的 6 大改进方向，构建 v3。

📚 六、业界资料与最佳实践

1. 上下文工程（Anthropic）

Anthropic 在 Effective context engineering for AI agents 中指出："Context engineering refers to the set of strategies for curating and maintaining the optimal set of tokens during LLM inference."

启示：不应把所有文档全量塞入 prompt，而应通过 RAG 检索只注入与当前资产相关的文档片段。

2. 分块策略（Pinecone / Atlan）

Pinecone 的 Chunking Strategies 和 Atlan 的 9 chunking strategies for RAG 都强调：不同内容类型应采用不同分块策略。对于表格类文档（Excel），应保留列名/数据类型作为 metadata；对于文本类文档（Word/PDF），应按语义段落分块。

启示：v1/v2 对文档的处理过于粗暴（直接拼接 45K 字符），应按文档类型分块 + 向量化 + 按需检索。

3. 结构化输出可靠性（SD Times / Cleanlab）

SD Times 在 LLM Output Reliability 中指出："Always validate with Pydantic or Zod, even when the provider guarantees schema compliance. Business logic validation catches what JSON Schema cannot."

启示：v1/v2 的 JSON 解析只有简单的 json.loads + fallback，缺少 Pydantic schema 验证和自校正重试。

4. 业务术语表 vs 数据字典（Alation / Atlan）

Alation 在 Business Glossary vs Data Dictionary 中区分了两个概念：数据字典（技术层：表/列/类型）和业务术语表（业务层：术语定义/业务规则）。Atlan 进一步指出："A business glossary teaches an AI agent what each term means while a catalog shows where the data lives."

启示：文档优化的核心价值是构建"业务术语表"——把文档中的业务术语映射到物理字段，而不是重新提取 fields。

5. 知识图谱增强（Neo4j GraphRAG）

Neo4j 在 Semantic Layer 中展示：用知识图谱替代静态 YAML schema 文件，可减少 20-30% token 消耗。GraphRAG 方式让 LLM "导航"元数据而非"阅读"全部元数据。

启示：Ontology 本身就是知识图谱，优化时应利用已有关系做"导航式检索"，而非全量注入。

6. 自动化元数据生成（arXiv）

arXiv 论文 Leveraging RAG for Automated Metadata Generation 探讨了用 RAG 自动为数据目录生成表级和列级描述的方法，核心是先检索相关文档片段，再让 LLM 基于片段生成描述。

启示：文档优化应采用 RAG 模式——先对文档分块向量化，优化每个资产时只检索相关片段。

💡 七、改进建议（6 大方向）

改进 1：文档 RAG 化（解决文档重复传输问题）

问题：v1/v2 每次都把文档全量拼入 prompt（45K 字符），多批次时文档被重复传输 N 次。

方案：

  • 文档上传时按语义分块（Word/PDF 按段落，Excel 按 sheet/行组）
  • 每块生成 embedding 存入向量库（如 Chroma/FAISS）
  • 优化每个 Class 时，用 Class 的 name_cn + description 做语义检索，只取 Top-K 相关文档片段
# 伪代码
for batch in batches:
    for cls in batch["classes"]:
        query = f"{cls['name_cn']} {cls['description']}"
        relevant_chunks = vector_store.search(query, top_k=5)
        cls_doc_context = "\n".join(relevant_chunks)
    # 每批只带相关文档片段，而非全量文档

收益 文档 token 消耗降低 80%+，且每个 Class 看到的文档更精准

改进 2：两阶段优化（解决全局视角缺失问题）

问题：v2 分批后 LLM 看不到全部资产，跨表关系推导和命名一致性无法保证。

方案：采用"先分后合"两阶段策略：

  • 阶段一（分批优化）：v2 的分批策略，每批优化 Class/Metrics/Concepts 的业务描述
  • 阶段二（全局校正）：把所有优化后的资产摘要（只含 id/name_cn/category，不含完整字段）送入 LLM，做全局一致性校正：
    • 统一命名风格（如所有 Class 名以"表"结尾）
    • 统一 category 分类标准
    • 推导跨表 Relationships
    • 补全 Concepts 树状层级
# 阶段二：全局校正
summaries = [{"id": c.id, "name_cn": c.name_cn, "category": c.category} for c in all_classes]
prompt = GLOBAL_CORRECTION_PROMPT.format(assets_summary=summaries)
corrections = llm.call(prompt)  # 只输出差异补丁，不是全量

收益 兼顾分批的上下文安全和全局的一致性

改进 3：Pydantic Schema 验证 + 自校正重试

问题：v1/v2 的 JSON 解析只有 json.loads + fallback，缺少结构验证和重试。

方案：

  • 用 Pydantic 定义所有资产的 schema（ClassSchema / MetricSchema / RelationshipSchema）
  • LLM 输出后用 Pydantic 验证，失败时把错误信息 + 原始输出回传给 LLM，让其修正
  • 最多重试 3 次
from pydantic import BaseModel, ValidationError

class MetricSchema(BaseModel):
    id: str
    name: str
    description: str
    target_class: str
    formula: str
    dimensions: list[str]
    required_dimensions: list[str]
    chart_type: str = "bar"

for attempt in range(3):
    raw = llm.call(prompt)
    try:
        metrics = [MetricSchema(**m) for m in raw["metrics"]]
        break
    except ValidationError as e:
        prompt += f"\n\n上次输出有误：{e}\n请修正后重新输出。"

收益 JSON 解析成功率从 ~70% 提升到 ~95%+

改进 4：增量优化模式

问题：v1/v2 每次都全量重跑，即使只新增了 1 个文档也要重新优化所有资产。

方案：

  • 记录每个资产的 last_optimized_at 和 source_docs（来自哪些文档）
  • 新增文档时，只优化与该文档语义相关的资产（通过向量检索确定）
  • 已审核（is_reviewed=1）的资产不自动覆盖，只标记"有新文档可优化"提示
# 增量优化
new_doc_chunks = embed_and_chunk(new_doc)
related_assets = vector_store.find_related(new_doc_chunks, assets)
# 只优化 related_assets，而非全量
batches = build_batches(related_assets)

收益 增量优化时 token 消耗降低 90%+，且不破坏已审核资产

改进 5：文档分类与结构化提取

问题：v1/v2 把所有文档当纯文本处理，丢失了 Excel 的表格结构和 Word 的标题层级。

方案：

  • Excel：按 sheet 分块，每块保留表头 + 数据类型 + 样本行，作为"结构化文档片段"
  • Word：按标题层级（H1/H2/H3）分块，保留层级路径作为 metadata
  • PDF：按页面/章节分块，保留页码和章节路径
  • 分块时生成 metadata：{doc_type, section_path, page, has_table}
# Word 文档结构化分块
{
    "doc_type": "word",
    "section_path": "第三章 > 3.2 销售指标定义",
    "content": "销售总额：所有门店所有品类的销售金额总和...",
    "metadata": {"heading_level": 2, "page": 15}
}

收益 文档片段携带结构信息，LLM 能更精准地理解业务上下文

改进 6：优化结果 Diff 审计 + 人工审核闭环

问题：v1/v2 直接 upsert 覆盖，用户看不到"改了什么"，且 is_reviewed 机制未形成闭环。

方案：

  • 优化后生成 Diff 报告：每个资产的 before/after 对比
  • Diff 报告存入 schema_optimization_runs.changes_json
  • 前端展示 Diff，用户可逐条 Accept / Reject
  • Reject 的资产保持原值，Accept 的资产标记 is_reviewed=1
# Diff 报告示例
{
    "classes": [
        {
            "id": "Sale",
            "field": "description",
            "before": "销售记录表",
            "after": "门店销售明细表，记录每笔交易的金额、品类、门店信息",
            "source_doc": "销售业务手册.pdf",
            "source_section": "2.1 销售数据定义"
        }
    ]
}

收益 优化过程可追溯、可审计，用户信任度高

✅ 总结

v2 的分批方向正确，但需补全代码并解决全局视角缺失问题。v3 蓝图通过"文档 RAG 化 + 两阶段优化 + Pydantic 验证 + 增量模式 + 结构化分块 + Diff 审计"6 大改进，可构建生产级 Schema 优化器。建议优先实现改进 1（文档 RAG 化）和改进 3（Pydantic 验证），这两项投入产出比最高。
